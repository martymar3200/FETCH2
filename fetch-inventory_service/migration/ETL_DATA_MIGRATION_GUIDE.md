# FETCH2 ETL Data Migration Guide

This guide outlines the procedure for migrating data from the legacy LC production database into a pristine FETCH2 production database using an ETL (Extract, Transform, Load) strategy.

Unlike an in-place schema merge, this approach guarantees a clean, technical-debt-free database by deploying a fresh FETCH2 schema and programmatically migrating the data over.

## Prerequisites

1. **Source Database:** Read-only access to the legacy LC database (`SOURCE_DB_URL`).
2. **Target Database:** Admin access to the new pristine FETCH2 database (`TARGET_DB_URL`).
3. **Python Environment:** The FETCH2 Python environment with all dependencies installed.

---

## Phase 1: Deploy the Clean FETCH2 Schema

Before moving any data, the new database must be built from scratch using the modern FETCH2 migrations.

1. Configure your environment to point to the **TARGET** database:
   ```bash
   export DATABASE_URL="postgresql://user:pass@host:5432/fetch2_prod_db"
   export MIGRATION_URL="postgresql://user:pass@host:5432/fetch2_prod_db"
   ```

2. Run the modern Alembic migrations:
   ```bash
   poetry run alembic upgrade head
   ```

3. The target database is now structurally perfect and completely empty.

---

## Phase 2: Execute the ETL Pipeline

We use a custom Python script to extract data from the legacy LC schema, transform it to fit the FETCH2 schema, and load it into the new tables.

1. Ensure the `SOURCE_DB_URL` environment variable is set:
   ```bash
   export SOURCE_DB_URL="postgresql://readonly_user:pass@lc_prod_host:5432/lc_prod_db"
   ```

2. Run the ETL script:
   ```bash
   poetry run python migration/run_etl_migration.py
   ```

3. **What happens during execution:**
   - The script will connect to both databases.
   - It will temporarily disable foreign key constraints on the target database to maximize bulk insert speed.
   - It transfers tables in strict dependency order (e.g., Users → Buildings → Trays → Items).
   - It handles schema transformations (e.g., remapping column names, translating Enums, and dropping unused legacy data like `audit_log`).
   - It restores constraints and validates row counts.

---

## Phase 3: Validation and Cutover

Once the script completes successfully:

1. Review the output logs generated by the ETL script to ensure `SOURCE` and `TARGET` row counts match for critical tables (Items, Trays, Users).
2. The FETCH2 database is now fully populated and ready for production deployment. The legacy LC database can be safely decommissioned.
